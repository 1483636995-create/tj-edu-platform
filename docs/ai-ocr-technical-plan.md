# AI/OCR 技术方案

## 目标范围

本方案用于规划天津教辅平台后续智能能力的服务边界和数据流，不在当前阶段直接接入具体 OCR、向量库或大模型供应商。第一版智能能力优先解决四类场景：

- 试卷、讲义、错题图片的 OCR 识别和结构化入库。
- 题目知识点识别、难度建议和错因标签建议。
- 基于题库、错题和讲义素材的相似题检索。
- 面向教师和教务的学生学习报告草稿生成。

## 服务边界

### 主业务 API

`apps/api` 是用户、权限、机构隔离和业务数据的唯一写入口。主业务 API 负责：

- 接收上传文件和错题图片元数据。
- 创建智能任务记录，维护任务状态、输入来源、发起人和审核状态。
- 向智能服务下发最小必要数据。
- 接收智能服务返回的候选结果。
- 将候选题目、候选知识点、候选错因和报告草稿进入人工审核队列。
- 只在审核通过后写入正式题库、错题本、讲义或学生档案。

### 智能服务

建议后续新增独立的 Python/FastAPI 服务，专注计算密集和模型编排：

- OCR 适配层：封装本地 OCR、云 OCR 或第三方 OCR。
- 版面解析层：识别题号、题干、选项、答案区、解析区和图片区域。
- NLP/LLM 编排层：做题目清洗、知识点候选、错因候选和报告草稿。
- 向量检索层：维护题目、知识点说明、讲义段落和错题摘要的 embedding。
- 风险过滤层：对模型输出做敏感信息、越权数据和低置信度拦截。

智能服务不能直接写主库业务表。它只能写自己的任务中间态，或通过主业务 API 回传候选结果。

### 异步队列

OCR 和大模型任务应全部异步化。建议沿用 Redis/BullMQ 作为主 API 的任务入口，智能服务消费任务或通过内部 HTTP 拉取任务：

- `ocr.extract`: 文件或图片识别。
- `question.parse`: OCR 文本转题目候选。
- `knowledge.match`: 题目或错题匹配知识点。
- `similarity.search`: 相似题召回。
- `report.draft`: 学习报告草稿生成。

任务状态至少包含 `PENDING`、`PROCESSING`、`NEEDS_REVIEW`、`APPROVED`、`REJECTED`、`FAILED`。

## 数据流

### 试卷/讲义 OCR 入库

1. 教师或教务上传文件到文件库。
2. 主 API 创建 OCR 任务，只传文件 ID、机构 ID、发起人 ID 和存储地址。
3. 智能服务读取文件，执行 OCR 和版面解析。
4. 智能服务返回题目候选、页码、置信度、原文片段和图片裁剪引用。
5. 主 API 写入候选表，状态为 `NEEDS_REVIEW`。
6. 教务在审核页面确认题干、答案、解析、学科、年级、知识点和来源。
7. 审核通过后正式写入 `Question`、`PaperQuestion` 和知识点关联。

### 错题识别

1. 学生端或教师端上传错题图片。
2. 主 API 创建错题识别任务，关联学生、课程或课节。
3. 智能服务识别题目文本、学生作答痕迹和可疑错误点。
4. 系统先尝试相似题召回，若命中题库题目，则生成 `Mistake` 候选。
5. 若未命中题库，则生成待审核题目候选和错题候选。
6. 教师审核后，错题进入学生错题本。

### 知识点匹配

1. 输入题干、答案、解析、学科、年级和来源。
2. 先使用规则过滤到同学科、同年级或通用知识点。
3. 使用 embedding 召回候选知识点。
4. 使用 LLM 或轻量分类器排序并给出原因。
5. 低置信度结果必须人工确认；高置信度也应保留可追溯解释。

### 相似题推荐

1. 将题目题干、解析、知识点、题型和难度生成向量。
2. 查询同机构可见范围内的题目：公共题库 + 当前机构题库。
3. 先按向量召回，再按学科、年级、题型、难度和知识点重排。
4. 返回相似题列表，不自动替换教师选择。

### 学习报告草稿

1. 主 API 聚合学生课程记录、错题状态、知识点掌握变化和教师备注。
2. 智能服务只接收生成报告所需的最小摘要，不接收无关个人信息。
3. 模型生成报告草稿，包括优势、薄弱点、下阶段建议和家校沟通摘要。
4. 教师必须审核编辑后才能对学生或家长可见。

## 数据模型预留

后续可新增以下表或等价模型：

- `AiTask`: 任务类型、状态、发起人、机构、输入引用、错误信息。
- `AiCandidateQuestion`: OCR/解析得到的候选题目。
- `AiCandidateMistake`: 错题识别候选。
- `AiReviewDecision`: 审核人、审核结果、修改差异和时间。
- `EmbeddingDocument`: 向量文档类型、来源 ID、模型版本、索引状态。
- `LearningReportDraft`: 学习报告草稿、生成依据、审核状态和发布状态。

正式业务表只保存审核通过的数据；候选表保存模型输出、置信度和原始引用，便于追溯和回滚。

## 人工审核

必须人工审核的内容：

- OCR 转成的新题目。
- 自动匹配的答案和解析。
- 低置信度知识点匹配。
- 错因标签和掌握状态建议。
- 对学生、家长可见的学习报告。

审核页面需要展示：

- 原始图片或文件页码。
- OCR 原文与结构化结果对照。
- 模型置信度和候选排序。
- 修改前后差异。
- 审核人、审核时间和驳回原因。

## 隐私与合规要求

平台服务对象包含未成年人，智能能力默认按高敏感度处理。落地前需要由法务或合规负责人确认适用要求，至少应覆盖《个人信息保护法》《数据安全法》《未成年人保护法》及教育主管部门的相关要求。

工程侧先按以下约束设计：

- 最小化：智能服务只接收完成任务必要的数据，优先传业务 ID 和脱敏摘要。
- 隔离：所有任务必须带 `institutionId`，查询和向量召回必须做机构隔离。
- 脱敏：模型输入不包含手机号、家庭住址、身份证件、家长联系方式等无关信息。
- 授权：涉及学生个人画像、报告生成和对外展示时，预留授权与撤回机制。
- 留痕：所有模型输出进入审核链路，保留任务 ID、模型版本、输入引用和审核记录。
- 可删除：学生、家长或机构要求删除数据时，业务数据、候选数据和向量文档要能级联处理。
- 供应商边界：使用外部 OCR 或大模型时，必须确认数据存储地点、训练使用约束、日志保留时间和删除机制。

## 技术选型建议

### OCR

第一阶段建议通过接口适配层同时支持本地 OCR 和云 OCR。选型时优先评估中文试卷、手写痕迹、表格、公式和图片裁剪能力。

### 向量库

开发期可先使用 PostgreSQL + pgvector 或独立轻量向量库。正式环境根据题量和并发再评估 Milvus、Elasticsearch 向量检索或云向量服务。

### 大模型

大模型只负责候选生成和草稿生成，不能直接写正式数据。提示词、模型版本和输出 schema 需要版本化，便于回归测试。

## 分阶段落地

### POC

- OCR 单文件识别。
- 题目候选结构化。
- 人工审核后写入题库。

### MVP

- 错题图片识别。
- 知识点候选匹配。
- 相似题召回。
- 学习报告草稿。

### 稳定化

- 批量任务、重试、死信队列和监控告警。
- 模型版本评估集。
- 审核效率指标。
- 供应商成本和准确率报表。

## 验收标准

- 文档明确主 API、智能服务、队列、对象存储、向量库之间的边界。
- 文档明确 OCR、错题识别、知识点匹配、相似题和学习报告的数据输入输出。
- 文档明确所有模型输出必须经过人工审核才能进入正式业务数据。
- 文档明确未成年人数据、机构隔离、脱敏、留痕、删除和供应商边界要求。
- 后续实现 PR 能按本文拆分为独立服务、任务表、候选表、审核 UI 和模型适配层。
